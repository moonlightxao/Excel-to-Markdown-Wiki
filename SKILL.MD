# Excel to Markdown Wiki 转换工具

## 工具描述

将包含故障运维数据的 Excel 文件自动转换为结构化的 Markdown Wiki 文件。
工具会解析 3 张关联工作表，通过外键关系组合数据，并调用 LLM 为每个故障场景生成完整的恢复预案。

## 调用命令

```bash
python main.py --full --excel <EXCEL_PATH> --output <OUTPUT_PATH>
```

### 参数说明

| 参数 | 必填 | 说明 |
|------|------|------|
| `--full` | 是 | 全量生成模式：生成故障现象/定界手段/恢复方案 Markdown + LLM 恢复预案 |
| `--excel <EXCEL_PATH>` | 是 | Excel 输入文件的绝对路径 |
| `--output <OUTPUT_PATH>` | 是 | 输出目录的绝对路径，所有生成的文件将写入此目录 |

> 注意：`--full` 和 `--sheets` 互斥。本工具使用 `--full` 模式以生成完整恢复预案。

## 输入格式

Excel 文件（.xlsx）必须包含以下 3 张工作表，Sheet 名称和列名需严格匹配：

### Sheet 1: 故障现象场景清单模版

| 列名 | 说明 | 示例 |
|------|------|------|
| 编号 | 故障现象 ID（UPxxxx 或 APxxxx） | UP0001 |
| 故障现象 | 故障名称 | 页面大量接口报错 |
| 现象描述 | 详细描述 | 用户访问页面时出现大量接口报错... |
| 定界方法 | 关联的定界手段 ID，多个用逗号分隔 | A0001,A0002 |
| 分类 | 故障分类（可选） | 网络类 |
| 故障感知手段 | 监控/告警方式（可选） | 监控平台告警 |
| 是否已有感知手段 | 是/否（可选） | 是 |
| 故障表现层级 | 层级标签（可选） | 用户界面层 |

### Sheet 2: 定界手段模板

| 列名 | 说明 | 示例 |
|------|------|------|
| 编号 | 定界手段 ID（Axxxx） | A0001 |
| 定界方法名称 | 方法名称 | 排查关键依赖平台是否异常 |
| 详细定界方法 | 具体步骤 | 1. 检查...\n2. 查看... |
| 恢复方案建议 | 关联的恢复方案 ID，多个用逗号分隔 | R0001,QR0002 |
| 定界工具 | 使用的工具（可选） | 命令行 |
| 定界结果 | 预期结果（可选） | 确认异常 |

### Sheet 3: 恢复方案模板

| 列名 | 说明 | 示例 |
|------|------|------|
| 编号 | 恢复方案 ID（QRxxxx 或 Rxxxx） | QR0001 |
| 应对方案名 | 方案名称 | 联系关键依赖平台支持人员协助处理 |
| 方案概述 | 具体步骤文本 | 1. 联系...\n2. 等待... |
| 恢复工具 | 使用的工具（可选） | 运维平台 |

### 外键关系

```
故障现象 (UPxxx/APxxx)
  └─→ 定界手段 (Axxxx)            通过「定界方法」列关联
        └─→ 恢复方案 (QRxxxx/Rxxxx)  通过「恢复方案建议」列关联
```

## 输出结构

执行成功后，`<OUTPUT_PATH>` 目录包含：

```
<OUTPUT_PATH>/
├── 故障现象/                          ← 每个故障现象一个 .md 文件
│   ├── UP0001_页面大量接口报错.md
│   └── AP0002_MQS消息重复.md
├── 定界手段/                          ← 每个定界手段一个 .md 文件
│   ├── A0001_排查关键依赖平台是否异常.md
│   └── A0002_排查数据库CPU使用率是否异常.md
├── 恢复方案/                          ← 每个恢复方案一个 .md 文件
│   ├── R0001_联系关键依赖平台支持人员协助处理.md
│   └── QR0002_数据库CPU异常恢复方案.md
├── UP0001_页面大量接口报错.md         ← LLM 生成的完整恢复预案
├── AP0002_MQS消息重复.md             ← LLM 生成的完整恢复预案
└── index.md                           ← 汇总索引
```

## 前置条件

### Python 依赖

```bash
pip install -r requirements.txt
```

依赖包含：`pandas>=3.0`、`openpyxl>=3.1`、`requests>=2.28`

### LLM 服务

必须有一个可用的 LLM 服务（默认为本地 Ollama）：

```bash
# 安装 Ollama
curl -fsSL https://ollama.com/install.sh | sh

# 启动服务
ollama serve

# 拉取模型（默认 qwen2.5:7b）
ollama pull qwen2.5:7b
```

LLM 服务需在 `http://localhost:11434` 可用。如需使用其他 LLM 服务（如 DeepSeek、通义千问），需修改 `config.py` 中的 `DEFAULT_CONFIG["llm"]` 配置。

### 常见错误

| 错误信息 | 原因 | 解决方法 |
|---------|------|---------|
| `错误: Excel 文件不存在` | 路径错误 | 检查 --excel 参数路径 |
| `✗ LLM 服务不可用` | Ollama 未运行 | 执行 `ollama serve` |
| `Model 'xxx' is not available` | 模型未拉取 | 执行 `ollama pull qwen2.5:7b` |
| `Sheet 'xxx' not found` | Excel Sheet 名称不匹配 | 检查 Excel 工作表命名 |

## 注意事项

1. 执行时间取决于故障案例数量和 LLM 速度，每个案例约需 10-60 秒
2. 多次运行会覆盖同名文件（增量覆盖策略）
3. 日志文件 `excel2md.log` 会记录在项目根目录
4. 如 Excel 列名与配置不完全匹配，工具会尝试模糊匹配并给出警告
